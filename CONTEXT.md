## What This Project Is

A research platform for studying automated UI revision. Two machine-learning pipelines:

1. **Revision Generator** — takes a Before screenshot + taxonomy category → generates a concrete revision task (what a designer/engineer should change and why).
2. **Evaluator** — takes a revision task + Before/After screenshot pair + DOM diff → outputs a 5-criterion rubric verdict (each PASS / PARTIAL PASS / FAIL) plus a binary OVERALL (PASS / FAIL) and a comment.

Both models were fine-tuned on Vertex AI (Gemini SFT) from labeled datasets of expert-reviewed UI revision examples.

## Dataset

**RawDataset** (`Datasets/RawDataset/`): Source examples, each a folder named `Participant_X_CaseStudy-Y.Z-MODEL` containing:
- `Before/index.html` + `Before/screenshot.png`
- `After/index.html` + `After/screenshot.png`
- `Task.txt` — the revision instruction
- `dom_diff.txt` — three-section computed DOM diff (see below)
- `step1_spec.txt` — UI component context generated by Step 1
- `Taxonomy.txt` — which of 7 categories this revision belongs to
- `Output.txt` — expert evaluation notes

**EvaluatorModelDataset** (`Datasets/EvaluatorModelDataset/`): 141 expert-labeled examples used for evaluator model training and testing. Same folder naming convention as RawDataset, but ground truth comes from `RubricEvaluation.json` (not `Output.txt`):
```json
{
  "criteria": {
    "requirementFulfillment": "pass",
    "consistencyOriginal":    "pass",
    "visualUsability":        "pass",
    "minimality":             "pass",
    "noRegressions":          "fail"
  },
  "overallEvaluation": "fail",
  "overallComment": "..."
}
```
Criterion values: `"pass"`, `"fail"`, `"partial"`, `"na"`.

**Naming convention**: `CaseStudy-Y.Z` means case study number Y (unique screen), participant variant Z. Dedup by extracting Y with regex `CaseStudy-(\d+)`.

**Model variants**: Each unique (participant, case study) combo was implemented by one or more models (CLAUDE, GEMINI, OPENAI). Model variants share the same source task and before screenshot.

**Train/Test splits**:
- EvaluatorModelDataset: 98 train / 43 test folders (50 / 23 groups), split at group level (no same-screen leakage), PASS/FAIL stratified, with guaranteed difficult-case inclusion in both train and test. Symlinked under `Train/` and `Test/`; manifest at `Datasets/EvaluatorModelDataset/split_manifest.json`.
- RevisionGeneratorModelDataset: ~80% / 20% (17 test examples), multi-label stratified by taxonomy category

## MUD_GenreUI Dataset

A 100-screen mobile UI benchmark used to generate a broader revision task dataset using the fine-tuned revision generator. Located at `Datasets/MUD_GenreUI/` and published to HuggingFace.

**Contents:**
- `images/` — 100 original mobile app screenshots (real app screens, 10 app types × 12 user intents)
- `html/` — 100 reconstructed HTML/CSS files (Gemini 2.5 Pro; phone status bars and nav bars excluded)
- `screenshots/` — Playwright renders of the HTML at 390px wide, full-page height
- `metadata/` — per-screen JSON metadata from the source MUD dataset
- `results_final_100.csv` — id, app_type, intent, ui_pattern, confidence
- `revisions.json` — `{ "<id>": { "applicable_categories": [...], "tasks": { "<category>": ["task1", "task2", "task3"] } } }`

**Pipeline (`MUD_Dataset_Utils/`):**

1. **HTML generation** (`MUD_generate_html.py`) — Gemini 2.5 Pro reconstructs each screenshot as a self-contained HTML/CSS file. Supports `--ids` to regenerate specific screens. Token config: `max_output_tokens=65536`, `thinking_budget=2048`.
2. **Screenshot rendering** (`MUD_screenshot.py`) — Playwright renders each HTML at 390px width, full-page height.
3. **Visual inspection** (`MUD_inspect.py`) — outputs `review_mud.md` with all rendered screenshots + metadata for quick review.
4. **Revision generation** (`MUD_generate_revisions.py`) — two steps:
   - *Filter*: Gemini 2.5 Pro (base) looks at the rendered screenshot and returns which of the 7 taxonomy categories have meaningful revision opportunities.
   - *Generate*: fine-tuned VertexAI generator (`VERTEXAI_GENERATOR_ENDPOINT_ID`) produces 3 tasks per applicable category.
   - Supports `--id <id>` for single-screen preview and `--backend gemini` to use base Gemini instead of fine-tuned model. Resume-safe.
5. **Revision inspection** (`MUD_inspect_revisions.py`) — outputs `review_mud_revisions.md` pairing each screenshot with its generated tasks.
6. **HuggingFace upload** (`MUD_upload_hf.py`) — packages dataset as `data/train.parquet` (one row per screen with embedded image bytes, screenshot bytes, HTML text, and revision tasks) plus raw HTML files. Supports `--token` or `HF_TOKEN` env var.

**Original screen curation** (`MUD_analysis.ipynb`): Gemini 2.5 Flash classified screenshots from a larger mobile UI dataset by app_type / intent / ui_pattern / usability, then selected a stratified 100-screen sample. The notebook also contains the initial single-screen HTML generation experiments.

## Evaluation Pipeline (Two Steps)

**Step 1** (`Evaluator/step1.py`): Given task + Before screenshot → generates a **visual attention guide** identifying which UI components to examine (location, appearance, relationships). This is a navigation aid for the evaluator, NOT an expected-change spec — it must not introduce implementation requirements beyond the task. Prompt in `Evaluator/step1_prompt.txt`. Results cached per-folder as `step1_spec.txt`.

**Step 2** (`Evaluator/step2.py`): Given task + Step 1 UI context + DOM diff + Before/After screenshots → outputs a structured rubric verdict. Prompt in `Evaluator/step2_prompt.txt`.

Output format:
```
REQUIREMENT FULFILLMENT: PASS / PARTIAL PASS / FAIL
CONSISTENCY: PASS / PARTIAL PASS / FAIL
VISUAL & USABILITY: PASS / PARTIAL PASS / FAIL
MINIMALITY: PASS / PARTIAL PASS / FAIL
NO REGRESSIONS: PASS / PARTIAL PASS / FAIL

OVERALL: PASS / FAIL

COMMENT: <1–3 sentence rationale>
```

Ablation flags (`--no-dom-diff`, `--no-step1`) can omit either optional input. Run all four ablation conditions at once with `Testing/Evaluator/run_ablations.sh`.

**DOM diff** (`Util/dom_diff.py`): Three sections computed via Playwright:
1. CSS Rule Changes — declared intent (`<style>` block diffs), may be overridden
2. Computed Style Changes — final browser-resolved values for elements matching changed selectors; authoritative for subtle changes (spacing, color, font)
3. DOM Structure Changes — structural additions/removals/attribute changes

The Step 2 prompt explicitly instructs the model to trust Computed Style Changes for subtle changes (spacing, padding, font size, color tweaks) where screenshots at mobile resolution cannot resolve pixel differences, and screenshots for broad visible changes (layout shifts, element add/remove, text content).

## Model Testing

### Evaluator Testing (`Testing/Evaluator/`)

```bash
# Full pipeline run
python Testing/Evaluator/run.py --backend gemini [--resume]
python Testing/Evaluator/run.py --backend vertexai --run-name finetuned-v1 [--resume]

# Ablation flags (omit DOM diff or Step 1 spec)
python Testing/Evaluator/run.py --no-dom-diff
python Testing/Evaluator/run.py --no-step1

# Run all 4 ablation conditions sequentially (saves to Results/ automatically)
bash Testing/Evaluator/run_ablations.sh

# Rerun only previously wrong examples from an earlier run
python Testing/Evaluator/run.py --rerun-failed Testing/Evaluator/Results/gemini-2.5-pro.json
# → saves to Results/gemini-2.5-pro-rerun.json

# Inspect results (wrong-first, with DOM diff + Step 1 in collapsible sections)
python Testing/Evaluator/inspect_results.py Testing/Evaluator/Results/finetuned-v1.json > review.md
```

Output JSON: `{ run_name, backend, model, no_dom_diff, no_step1, examples: [{folder, ground_truth, gt_criteria, predicted, pred_criteria, comment, response}], metrics: {accuracy, macro_f1, per_class, confusion_matrix} }`

### Revision Generator Testing (`Testing/RevisionGenerator/`)

**run.py** — generates 3 tasks × 7 categories for each of the 11 unique test screens (77 total per run):
```bash
python Testing/RevisionGenerator/run.py --backend gemini --run-name gemini [--resume]
python Testing/RevisionGenerator/run.py --backend vertexai --run-name finetuned-v1
```

Output JSON: `{ run_name, screens: { "13": { example_key, source_folder, categories: { "Category Name": ["task1", "task2", "task3"] } } } }`

**compare.py** — pairwise A/B comparison using Claude as judge (avoids same-model bias):
```bash
python Testing/RevisionGenerator/compare.py \
    Testing/RevisionGenerator/Results/gemini.json \
    Testing/RevisionGenerator/Results/finetuned-v1.json \
    [--batch-size 0]   # 0 = all 7 categories per screen in one call (default)
    [--resume]
```

- A/B slot randomized once per screen (same flip for all categories in that screen's batch)
- Judge prompt shows all categories for a screen in one call; response parsed by `[CATEGORY: name]` markers
- Output JSON: `{ run_a, run_b, judge_model, summary: { totals, avg_scores, per_category }, comparisons: [...] }`
- `avg_scores`: per-run averages for Relevance, Specificity, Actionability, Diversity, overall (out of 5)

**add_avg_scores.py** — backfills `summary.avg_scores` into existing comparison files:
```bash
python Testing/RevisionGenerator/add_avg_scores.py Testing/RevisionGenerator/Results/gemini_vs_finetuned-v1.json
```

## Important Design Decisions

- **No same-screen leakage in splits**: train/test split is at the group level (unique `CaseStudy-Y.Z`). Model variants of the same case study always go to the same split.
- **Step 1 as attention guide, not spec**: Step 1 identifies which UI components to look at (location, appearance) — it must NOT prescribe how they should change. The block label in the Step 2 prompt is "UI Component Context (visual navigation aid — not additional requirements)" to prevent the model from treating it as requirements.
- **Step 1 deduplication**: Step 1 spec is generated once per unique `Participant_X_CaseStudy-Y.Z` key (ignoring MODEL suffix) and shared across all model variants.
- **Rubric output format**: The evaluator outputs 5 criteria (Requirement Fulfillment, Consistency, Visual & Usability, Minimality, No Regressions) each as PASS / PARTIAL PASS / FAIL, plus a binary OVERALL PASS / FAIL and COMMENT. PARTIAL PASS is not allowed at the overall level.
- **Difficult-case split**: `split.py` explicitly forces a configurable fraction of "difficult" groups (those where the base model was wrong) into training AND test separately before running the stratified split on the remainder — ensuring both fine-tuning and evaluation cover hard cases.
- **Computed styles over declared styles**: The DOM diff includes Playwright-computed styles because declared CSS can be overridden. The evaluator is instructed to use computed styles for subtle changes.
- **Inline style detection**: Position-path based matching (tag + sibling index, no classes) catches cases where inline `style=` is removed and a utility class is added simultaneously.
- **Claude as judge**: The revision generator comparison uses Anthropic Claude (not Gemini) as judge to prevent same-model favoritism.
- **Batched comparison calls**: compare.py sends all 7 categories for a screen in one API call (11 calls total) since they share the same screenshot.
