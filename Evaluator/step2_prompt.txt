**On flexible task language:**
When the task uses "or," "e.g.," "such as," or "for example," it is offering options or examples — not prescribing one specific approach. Evaluate against the stated intent, not against one particular example. A different but equally valid implementation that achieves the same goal should pass.

**On partial implementations:**
If the core intent of the task was accomplished and only secondary details are missing, lean toward an overall PASS rather than FAIL. This applies to missing minor details, not to unrequested changes that broke the design.

For tasks with multiple explicit instructions (listed separately or described in multiple sentences), verify that each instruction was addressed. Missing a single minor detail can still be PARTIAL PASS, but missing a complete sub-task — for example, the instruction said "on both sides" but only one side changed, or "above the entire group" but the label was added to sub-sections instead — should lower Requirement Fulfillment to PARTIAL PASS or FAIL, not PASS.

**On the DOM diff as ground truth:**
The DOM diff is the most reliable signal for what actually changed in the code. Use it as the primary evidence for any claim about additions, removals, or modifications:
- **Computed Style Changes** are browser-authoritative: if a property appears there, the change is confirmed even if it looks subtle in screenshots. If this section reports a removal (e.g. a background color going to `transparent`), treat it as a real regression or fulfillment signal.
- **Before asserting that something was removed or changed**, verify it appears in the DOM diff. If the screenshots seem to show a difference but neither the DOM Structure nor Computed Style sections corroborate it, do not assert the change occurred — mobile screenshot resolution can create false impressions.
- **Before asserting something was NOT changed**, check both sections. A property can change via a CSS class (not visible in DOM Structure) but show up in Computed Styles. Conversely, a very subtle change may be real but visually imperceptible.

**On subtle visual changes:**
If the screenshots appear unchanged for a task involving text corrections, minor color tweaks, or small CSS adjustments, check the DOM diff carefully before concluding no change was made — small changes are often invisible at screenshot resolution.

Exception for text-only corrections: if the task is to fix a typo, rephrase specific words, or correct text content, text node changes may not appear in any section of the DOM diff. In this case, rely on the Before/After screenshots as the primary signal — if the corrected text is visible in the After screenshot, the task was completed.

**On regressions — actively check for these:**
Before scoring No Regressions, explicitly look for:
- New empty or blank areas where content used to be or where the layout now looks unbalanced — pay special attention to blank gaps created when elements are moved out of containers
- Elements visually clipped or cut off at the screen edge
- Overlapping elements that obstruct each other
- Unrequested changes to elements not mentioned in the task (moved, restyled, or removed) — including elements that shifted to a new line or were repositioned. Check the DOM diff for any element that changed but was not part of the requested task; if its change has visible impact (different position, missing background, relocated in the layout), score Minimality or Consistency as FAIL even if the primary requirement was met.
- Excessive new whitespace or padding that makes the layout feel broken or unbalanced
- Content changes (text altered, removed, or added beyond the task scope)

Note: do not penalize for implicit layout side-effects not requested in the task (e.g. automatic reflow after an element is removed) — only penalize if the result is visually broken or clearly worse.

When many elements show changed `top`/`left`/`right`/`bottom` values in Computed Style Changes but the DOM Structure shows only one element was added, removed, or resized, this is layout reflow from a single change — treat it as one event, not as multiple individual regressions. Focus on whether the final visible layout is broken, not on how many computed property entries changed.

If a task requires removing or changing a CSS property and an adjacent or coupled property on the same element changes as a result (e.g., removing `border-left` also adjusts `padding-left` on the same element), treat this as a natural side-effect of the primary fix — not a separate regression — unless the visual result is clearly worse.

Does the revision avoid introducing new layout, visual, content, or interaction problems elsewhere in the interface (e.g. overlapping elements, broken alignment, clipped content, removed sections)?

The overall verdict is binary — PASS or FAIL only (no PARTIAL PASS at this level). Use judgment:
