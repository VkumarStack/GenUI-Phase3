EVALUATION NOTES:

**On the code analysis:**
The Code Analysis above summarizes what the developer's code changes do and flags any potentially unrelated or problematic changes. Use it as supplementary context — not as a binding verdict:
- Let it direct your attention to specific areas of the After screenshot for confirmation. In particular, go through each item in the Completeness Check and verify it visually — do not assume a requirement was met just because the code analysis did not flag it as missing.
- For changes that are subtle and hard to verify visually (minor padding shifts, small color tweaks, font size adjustments, text corrections), rely more heavily on the code analysis — if the diff clearly shows a change was made, trust that it happened even if it is hard to see at screenshot resolution.
- **Exception — icon identity:** When the task involves changing or replacing an icon, verify visually that the icon in the After screenshot actually looks like the intended icon. A class name or attribute change in the code does not guarantee the correct icon rendered — use the screenshot to confirm the icon's appearance matches what was requested.
- Do not let the code analysis override clear visual evidence. If the After screenshot contradicts what the code suggests, the screenshot is the ground truth for what the user actually sees.
- When the code analysis flags an unrelated or potentially problematic change, verify its visual effect in the screenshots before scoring it as a regression. Not every unrelated code change produces a visible problem.

**On flexible task language:**
When the task uses "or," "e.g.," "such as," or "for example," it is offering options or examples — not prescribing one specific approach. Evaluate against the stated intent, not against one particular example. A different but equally valid implementation that achieves the same goal should pass.

**On partial implementations:**
If the core intent of the task was accomplished and only secondary details are missing, lean toward an overall PASS rather than FAIL. This applies to missing minor details, not to unrequested changes that broke the design.

When a task explicitly names specific visual elements as part of a coordinated redesign (e.g., "use three dots for edit," "make it a rectangular button," "add a View Pass button"), each named element is a required deliverable, not an optional detail. If one of the explicitly named elements is absent or replaced with a clearly different alternative, this constitutes a missing sub-task and should result in FAIL on Requirement Fulfillment.

When a task applies to a repeating UI pattern — such as "each card," "all posts," "every button of this type" — the fix must be applied consistently to all instances that were visible in the Before screenshot. Only instances that were already rendered and visible before the revision can reasonably be expected to have been updated. Fixing only a subset of the visible instances is a significant omission and should typically result in an overall FAIL. Do not penalize for instances that were not present or not visible in the Before screenshot.

Important: the Before screenshot represents the complete set of elements that exist in the underlying HTML. If an element appears partially cut off at the bottom edge of the viewport, that is simply where the rendered content ends — it does not imply there is hidden or off-screen content in the HTML that also needed updating. Treat the visible screen as a complete and faithful representation of what the HTML contains. Do not penalize for "missing" instances that are only partially visible at the viewport boundary.

Omission vs. reversal: there is a critical difference between *not implementing* a requirement and *actively worsening* it. If the code analysis or screenshots show that the revision made a named requirement worse (e.g., the task said "fix text that's cut off" but the diff shows `text-overflow: ellipsis` was added, enforcing truncation), this is not a minor omission — it is an active contradiction of a named sub-task. Score Requirement Fulfillment as FAIL for that sub-task, and do not allow strong performance on other sub-tasks to mask a direct reversal of a stated goal.

**On move tasks — addition and removal are both required:**
When the task describes moving an element from one location to another, a complete implementation requires both (1) adding the element at the new location AND (2) removing it from the original location. If the code analysis shows only an addition, check the After screenshot to confirm the element is no longer visible in its original location. If the element appears in both locations after the revision, the move is incomplete — treat this as PARTIAL PASS or FAIL on Requirement Fulfillment and as a regression under No Regressions (duplicate content). If there are any lingering elements from a removal task, that should also count as a FAIL since the task was not fully completed.

When the task specifies moving an element to a particular location, use the code analysis to understand where in the HTML the element landed. Verify in the After screenshot that the element visually appears where the task specified — an element being nested inside another container when the task said to place it below or beside that container is an incorrect placement, even if it appears visually close.

**On subtle visual changes:**
If the screenshots appear unchanged for a task involving text corrections, minor color tweaks, or small CSS adjustments, consult the code analysis before concluding no change was made — small style changes are often invisible at screenshot resolution. If the code analysis confirms the change was made, treat the task as completed for that aspect.

**On regressions — require evidence, do not hallucinate:**
A regression must be grounded in concrete evidence — do not infer problems from screenshots alone. Only flag a regression if it meets one of these two bars:

1. **Code analysis evidence**: The code analysis flags a change that was not required by the task AND that change has a clearly visible negative effect in the After screenshot (broken layout, missing content, overlapping elements, clipped content).
2. **Unambiguously visible in both screenshots**: The after screenshot shows an obviously broken or degraded layout that is not present in the before screenshot — something that no reasonable reviewer could miss and that is clearly not intended.

**Regression severity — FAIL vs PARTIAL PASS on No Regressions:**
Not every confirmed regression warrants a FAIL. Use this threshold:
- Score **FAIL** only when the regression is *majorly noticeable* (immediately obvious without close inspection) AND *actively makes the user experience worse* — e.g. an element is completely missing, content is unreadable, the layout is broken or unusable, or a key interactive element was removed.
- Score **PARTIAL PASS** for regressions that are visible but do not significantly impair usability — e.g. an element is slightly misaligned, an aesthetic property changed unintentionally (background stretches wider, a container loses tight wrapping), or spacing is off in a way that looks awkward but keeps the interface functional and readable.
- An aesthetically imperfect but functional side-effect of an otherwise valid implementation is a PARTIAL PASS concern, not a FAIL.

Do NOT flag as a regression:
- Positional shifts of elements not flagged in the code analysis (this is layout reflow from an adjacent change — expected browser behavior)
- Visual differences that are subtle, ambiguous, or could be natural reflow from the requested change
- Anything that looks slightly different at screenshot resolution but is not corroborated by the code analysis
- Coupled property side-effects on the same element (e.g. adjusting one side of a border affecting adjacent spacing — this is a browser behavior, not a regression)

When only one element changed in the code but the layout shifted in the screenshots, that shift is reflow. Score it as a regression only if the final layout is clearly broken or unusable — not merely different.

**Specific regression patterns to check:**

*Unrequested removal of interactive elements:* If the code analysis flags that an interactive or functional UI element (button, menu, icon with an action) was removed and that element was not mentioned in the task as something to remove, verify in the After screenshot that it is indeed gone. Unrelated removals should lean towards overall FAILURE.


*Unrequested property deletions:* If the code analysis flags a CSS property deletion that was not required by the task (e.g., `width: 100%` removed when the task was only to change a color), and the deletion causes a clearly visible layout effect (element shrinks, shifts, or loses alignment), flag it under Minimality and No Regressions.

*Orphaned dependents after removal:* When the code analysis flags an element removal, check whether sibling or child elements that depended on it for positioning or compositional context became displaced or visually broken. This is distinct from layout reflow — if a badge, overlay, or icon now appears in the wrong location because the element anchoring it was removed, that is a regression.

*Container dissolution and de-nesting:* When the code analysis flags elements being de-nested or a container being dissolved, compare before and after screenshots to check whether elements that previously appeared in the same row or visual group now appear on a different line or in a broken position. If the code change caused this, it is a regression — not reflow.

*Large blank areas:* Before concluding there are no regressions, compare the vertical space distribution between before and after screenshots. Substantially more blank or empty space in a content area — particularly where content previously existed — is a visible layout regression.

*Named UI patterns must be complete:* When the task requests converting an element to a recognizable pattern (e.g., "outline button," "pill badge," "card"), verify in the after screenshot that the result actually looks like a complete instance of that pattern. If associated elements (like an icon paired with button text) are visually outside the styled region, the component is only partially converted.

Evaluate the revision using the rubric below. Score each criterion as PASS, PARTIAL PASS, or FAIL, then give a single overall verdict of PASS or FAIL.

**A. Requirement Fulfillment** *(most important criterion)*
Did the revision successfully perform the requested UI change? Consider whether the specific task was addressed, the correct element(s) were modified, and all parts of the instruction were followed. For move tasks, verify both that the element was added at the destination AND removed from the source. For tasks describing a specific visual outcome, verify the After screenshot confirms it — not just that the code changed.

**B. Consistency with Original UI**
Does the revision preserve the original screen's layout, structure, and visual design language? Unrelated sections should remain intact and the revised UI should feel like the same app/screen.

**C. Visual & Usability Quality**
Does the revision improve clarity, visual hierarchy, readability, accessibility, or overall usability — not just technically apply the change?

**D. Minimality of Changes**
Does the revision avoid unnecessary or unrelated modifications beyond what was requested? Use the code analysis to check scope: if the task targets one element, were any sibling or parent elements also modified beyond what is mechanically necessary? Note: even when modifying the correct target element, deleting CSS properties that were not required by the task counts as an unnecessary modification if it produces a visible effect.

**E. No New Regressions**
Does the revision avoid introducing new layout, visual, content, or interaction problems elsewhere in the interface (e.g. overlapping elements, broken alignment, clipped content, removed sections)? Check each specific regression pattern listed above before scoring this criterion.

OVERALL VERDICT GUIDANCE:
The overall verdict is binary — PASS or FAIL only (no PARTIAL PASS at this level). Use judgment:
- Requirement Fulfillment is the most critical criterion. A FAIL on the core task typically results in an overall FAIL, even if other criteria pass.
- A PARTIAL PASS on Requirement Fulfillment that accomplished the primary intent of the task often still warrants an overall PASS — use judgment on whether what was missing was central to the designer's goal or a secondary detail.
- However: when a task applies to multiple equivalent/repeating elements and the fix was only applied to some of them, this is a central omission and should typically result in an overall FAIL. Similarly, when a named sub-task was actively reversed (the change made it worse), this is a central failure — not a secondary detail.
- A PASS on Requirement Fulfillment with minor isolated issues elsewhere can still be an overall PASS.
- A critical regression can override an otherwise passing score even when Requirement Fulfillment is met — but only when the regression is severe: an entire UI section removed, the interface is rendered unusable or unreadable, or a major content element is lost. Aesthetic side-effects (an element wider than intended, spacing slightly off, a container losing tight wrapping) are not severe enough to override a passing Requirement Fulfillment score.
- If the revision made unrequested changes with visible impact — beyond natural layout reflow from the requested change — consider whether the net result is coherent or whether the added changes compromise the design. Unrequested removal of interactive/functional elements (buttons, menus) or unrequested deletion of structural layout properties (e.g., `width: 100%` causing an element to collapse) are meaningful unrequested changes that can justify an overall FAIL even when the primary requirement was met.
- The overall verdict should reflect whether the revision was a net positive toward the designer's intent — not a mechanical average of criterion scores.
